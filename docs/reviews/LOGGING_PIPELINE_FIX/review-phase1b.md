## Findings

### 1. [minor / smell] `deploy.sh` unconditionally clobbers `.env`
`deploy.sh:18-21`

The heredoc rewrites `.env` from scratch on every deploy. Today `.env` only holds `HOST_UID`/`HOST_GID`, so this is harmless and matches the intent stated in the inline comment ("Overwritten on every deploy so a re-deploy by a different host user gets fresh values"). However, if anyone later parks unrelated runtime config in `.env` (Mongo URI overrides, secret tokens for a future feature, etc.), this script will silently drop them. Since `.env` is now `.gitignore`'d, the loss would be uncommitted-only and hard to detect.

Suggested fix shape: either (a) keep clobber-on-deploy but split the docker-compose env file into two — `.env.deploy` (managed by `deploy.sh`, UID/GID only) and `.env` (operator-owned) — referenced with `env_file:` instead of `.env` auto-loading; or (b) leave as-is and add a one-line comment in the top of `.env` like `# Auto-generated by deploy.sh on every run. DO NOT EDIT.` so the next operator sees it before editing. (b) is the lower-effort option and probably sufficient.

Not a blocker.

### 2. [minor / smell] No defensive default on `${HOST_UID}`/`${HOST_GID}`
`docker-compose.yml:18`

`user: "${HOST_UID}:${HOST_GID}"` correctly relies on the absent-var-fails-loud-at-`up`-time behavior (matches plan AD 2). I verified that no `${HOST_UID:-0}` default is present — good, because a default of `0` would silently run the container as root, which is exactly the failure mode we don't want. Worth calling this out explicitly in a comment above the line so a future contributor doesn't "helpfully" add a default. Pure documentation nit.

Suggested fix shape:
```yaml
# user: deliberately left without ${VAR:-default} so an unset HOST_UID/HOST_GID
# fails `docker-compose up` rather than silently running as root. Set via
# .env, which is written by deploy.sh from `id -u`/`id -g`.
user: "${HOST_UID}:${HOST_GID}"
```

Not a blocker.

## Verified (looks good)

- **`.env` heredoc semantics.** `cat > .env <<EOF` with **unquoted** `EOF` delimiter — host-shell substitution of `$(id -u)` and `$(id -g)` happens before the redirect, so the file contains literal numeric UID/GID. Confirmed by reading `deploy.sh:18-21`. If `EOF` had been quoted (`<<'EOF'`) substitution would have been suppressed and we'd write the literal string `$(id -u)`. The unquoted form is correct here.
- **`user:` placement.** Added to `bot-manager` service only (`docker-compose.yml:18`). `mongo`, `loki`, `promtail`, `grafana` correctly untouched — they continue to run as their image defaults. `promtail` in particular needs `root` (or appropriately privileged) to read the bind-mounted log dir; leaving its `user:` unset is correct.
- **`.env` file permissions.** Default umask 0022 → 0644 on the generated `.env`. Fine for UID/GID values, which are not secrets. (Caveat from finding 1: this becomes a concern only if secrets ever creep into the file.)
- **`deploy.sh` sudo/chown/chmod removal.** Confirmed: no `sudo`, `chown`, `chmod`, `stat`, or `usermod` in the new `deploy.sh`. Privileged ops are completely gone from the deploy path.
- **`deploy.sh` cwd anchor retained.** `cd "$(dirname "$(readlink -f "$0")")"` at line 6 is intact (Phase 1 polish carry-over).
- **`set -euo pipefail`** retained in both `deploy.sh:2` and `scripts/host-prep-logs.sh:23`.
- **`host-prep-logs.sh` ordering.** `mkdir -p "$LOGS_DIR"` (line 30) runs **before** the `ls -A` stale-file scan (line 33). On a fresh host this means the scan runs against an empty (just-created) directory and the `if` body is skipped — correct. Previously the diff swapped to `[ -n "$(ls -A ...)" ]` without the `[ -d ... ]` guard, which would have been unsafe without the preceding `mkdir`; with the new ordering, it's safe.
- **Root refusal removed.** `host-prep-logs.sh` no longer refuses to run as non-root (the inverse check is also gone, i.e. it doesn't refuse to run as root either). Plan said "decide" — Dev chose "no refusal." That's defensible: running this script as root still works (root can `mv` polkitd files even more easily than world-writable-bit reliance), so a permissive stance keeps the script useful for either operator profile.
- **Backup target timestamp precision.** `BACKUP_DIR=".../logs.bak.$(date +%Y%m%d-%H%M%S)"` (line 28) — second-precision per Phase 1 polish. Same-day re-runs won't collide.
- **Idempotency / re-run on clean dir.** Walked through: `mkdir -p` no-ops; `ls -A` on an empty dir returns empty string → `[ -n "" ]` is false → the `mv` block is skipped → `BACKUP_DIR` is never created (mkdir is inside the `if`). Final `ls -la` prints. Clean re-run = no side effects. Good.
- **World-writable bit assumption.** Read the script carefully — there's no `test -w`, no `stat`, no permission check anywhere. The script relies solely on the OS letting `sgame` `mv` files inside a directory `sgame` owns (or that is world-writable). Either condition suffices; the script doesn't care which. Correct.
- **`.gitignore` placement.** `.env` entry at lines 44-45, in its own `### Deploy-generated env file ... ###` section, sits alongside the other `###` grouping convention. Not duplicated. Reasonable spot.
- **Scope.** `git diff e0da939..6ee86f0` outside the four files in the plan returns empty. No Dockerfile, log4j, pom.xml, or Java source touched. No scope creep.
